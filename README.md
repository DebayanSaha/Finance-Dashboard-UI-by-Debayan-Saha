#  Finance Dashboard UI

##  Overview

This project started with a clear goal: build a personal finance dashboard that feels genuinely useful — not just a collection of charts, but a cohesive tool that gives users a real sense of their financial health at a glance. The result is a clean, modern web application for tracking financial activity, built entirely on robust frontend fundamentals. Users can explore an overview of their finances, manage transactions, and dig into detailed spending insights — all securely within the browser, with zero backend required.

---

##  Requirement Mapping

Every feature was developed with direct traceability back to the assignment requirements:

*   **Dashboard Overview:** Implemented dynamic KPI cards (Balance, Income, Expenses), a time-based balance trend chart, and a categorical spending breakdown chart.
*   **Transactions Section:** Built a comprehensive transaction list with search, category filtering, and sorting capabilities.
*   **Role-Based UI:** Integrated a simulated frontend permission system allowing seamless switching between *Viewer* (read-only) and *Admin* (read/write) roles.
*   **Insights Section:** Added high-level analytics surfacing the highest spending category, monthly comparisons, and detailed breakdowns.
*   **State Management:** Utilized React Context API for global, scalable management of transactions, filters, and user roles.
*   **UI/UX Expectations:** Delivered a fully responsive, polished interface with thoughtful empty-state handling across all views — filtered or empty data is always handled gracefully, never leaving a broken or blank screen.

---

##  Features

*   **📱 Dashboard:** A centralized hub that gives users an immediate visual snapshot of their financial health through KPI cards and interactive, time-filtered charts.

*   **💸 Transactions Management:** An interactive ledger for browsing and managing transaction history with ease.
    *   **Data Structure:** Each transaction carries structured fields — Date, Amount, Category, and Type — ensuring consistent data across the app.
    *   **Interactivity:** Supports real-time search, category filtering, and chronological sorting that respond instantly as the user types or selects.
    *   **Access Control:** Admin users get full CRUD access — create, read, update, and delete — while Viewers are limited to reading, keeping the experience appropriately scoped to each role.

*   **📈 Deep Insights:** Detailed visual breakdowns of spending patterns, categorizations, and — where applicable — geographical transaction data.
    *   **How Insights are Computed:** All metrics are dynamically derived from raw transaction data. "Highest Spending Category" is resolved through category aggregation, and "Monthly Comparisons" are generated by grouping transactions by month on the fly. Nothing is hardcoded, so the insights stay perfectly in sync with the data at all times.

*   **⚡ Real-Time Updates:** Adding or editing a transaction immediately propagates across balances, charts, and insights — the entire UI stays synchronized without any manual refresh.

*   **🔐 Simulated RBAC:** A built-in role switcher demonstrates conditional UI rendering based on user capability, making frontend access control tangible and easy to evaluate.

---

##  Role-Based UI Behavior

To demonstrate frontend access control, a simulated Role-Based Access Control (RBAC) system is implemented. The goal was to show how capability-driven UI can be managed cleanly at the frontend layer:

*   **Admin Role:** Full read and write permissions. Admins can view all data, add new transactions, edit existing ones, and delete records safely.
*   **Viewer Role:** Strictly read-only access. When switched to Viewer mode, all forms, "Add Transaction" buttons, and edit/delete actions are hidden or disabled — not just visually, but logically excluded from the rendered output.
*   **Implementation:** Role switching is handled via a global toggle that updates application-wide state and dynamically adjusts the UI across all pages — no page reloads, no flicker.

##  Tech Stack & Architecture

Each technology in this stack was chosen deliberately, not by default:

*   **React:** Its component-based architecture makes it the natural fit for an application of this complexity — reusable, testable pieces that compose cleanly into larger views.
*   **Vite:** Chosen for its near-instant HMR and optimized production builds, which kept the development loop tight and fast throughout the project.
*   **React Context API:** Used for state management to eliminate prop-drilling and establish a single, predictable source of truth for transactions and roles — without the overhead of Redux for a project of this scope.
*   **Recharts:** Integrates seamlessly with React and makes it easy to build responsive, declarative charts that adapt to the data without manual DOM manipulation.
*   **CSS / Styling:** Custom CSS was used throughout to demonstrate strong core fundamentals — design token management, responsive layout control, and deliberate spacing — without leaning on utility frameworks as a shortcut.

---

##  Edge Case Handling & Performance

Robustness was treated as a first-class concern, not an afterthought:

*   **Empty States:** When a user clears all data or applies filters that return no results, clean and informative "No Transactions Found" placeholders appear. The layout never breaks or goes blank.
*   **Invalid Input Safety:** Form handling validates submissions before they hit state, preventing malformed or incomplete transaction records from entering the dataset.
*   **Data Persistence Fallback:** The app initializes safely even if `localStorage` is completely empty on first load — there's no assumption that persisted data exists.
*   **Performance Considerations:** Common array operations (filter, sort) are applied efficiently to keep interactions snappy. Expensive aggregations — such as chart data generation or monthly grouping — use optimized calculation patterns to maintain UI responsiveness even as the dataset grows.

##  Responsiveness

The dashboard is built to work on any device without compromise:

*   **Desktop & Tablet:** Expanded layouts maximize data density and allow side-by-side chart comparisons for a more analytical view.
*   **Mobile:** Gracefully collapses into a single-column, scrollable layout with an accessible, mobile-friendly navigation system. Charts automatically resize to fit smaller viewports without overflow or clipping.

##  Key Engineering Takeaways

A few core frontend engineering principles shaped every major decision in this project:

*   **Derived State Over Stored State:** Complex metrics are calculated on the fly rather than stored redundantly. This eliminates an entire class of data desynchronization bugs and keeps the codebase leaner.
*   **Single Source of Truth:** A unified state layer via Context API ensures predictable, traceable data flow — no ambiguity about where a value comes from or where it updates.
*   **Simulated RBAC:** Capability-based UI gating, managed purely on the frontend, demonstrates how access control logic can be expressed clearly through conditional rendering without backend involvement.
*   **Simplicity Meets Scalability:** The architectural patterns here deliberately mirror production-grade standards — not because the project demands it at this scale, but because good habits compound.

##  Project Structure

The codebase is organized around separation of concerns:

*   `src/components/` → Modular, reusable UI components (Cards, Charts, Forms, etc.)
*   `src/context/` → Global state management — the single source of truth for transactions and roles
*   `src/utils/` → Reusable helpers for currency formatting, data derivation, and aggregations
*   `src/pages/` → Top-level route containers that assemble components into full views (Dashboard, Transactions, Insights)

---

##  State Management & Data Logic

The application's data model is deliberately simple and strict:

*   **Single Source of Truth:** Only raw transactions are stored in the global Context. Everything else — balances, totals, breakdowns — is derived from that one list.
*   **Derived Data over Stored Data:** Metrics like "Total Balance" or "Total Income" are never stored as independent state variables. They're computed dynamically whenever the transaction list changes. This design choice is what keeps all views perfectly in sync without any manual reconciliation.
*   **localStorage Integration:** Inside the Context provider, changes to the transaction list are automatically intercepted and serialized into `localStorage`, giving the app zero-backend persistence that survives page reloads and browser closures.

##  Additional Enhancements

Beyond the core requirements, several quality-of-life improvements were added to round out the experience:

*   **Data Persistence (localStorage):** User data survives page reloads and browser closures without any backend dependency.
*   **Dynamic Chart Filtering:** Users can toggle chart views between Day, Month, and Year granularities to explore their data at different levels of detail.

*   **Dark Mode Support:** The design system extends cleanly into dark mode, maintaining aesthetic consistency across both themes.

##  Future Improvements (If Given More Time)

Given more runway, here's where this project would go next:

*   **Backend & Authentication:** Move away from `localStorage` toward a proper Node.js/PostgreSQL backend with JWT-based authentication and server-side session management.

*   **Advanced Filtering:** Add multi-select tags and complex date-range query filters for deep financial auditing across longer time horizons.
